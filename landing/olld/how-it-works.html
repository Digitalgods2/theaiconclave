How it works

Four phases.
One verdict.

The chamber separates deliberation from execution. Reasoning happens in one place, with rules. Execution happens elsewhere, with you in charge. That boundary is the product.

01
The question

Pose what you'd otherwise decide alone.

The AI Conclave is built for moments where a single confident answer would be dangerous. Architecture choices. Security tradeoffs. Refactors that could ripple. Policy drafts.

The chamber accepts the question along with whatever context fits: a code diff, a doc, an image, a spec. The session inherits prior decisions from your project, so the council isn't starting cold.

  • Project memory injected automatically
  • Provenance recorded (CLI, dashboard, API, Claude Code)
  • Read-only sandbox bound to the working tree

02
The council

Pick the seats per task.

Not every question deserves the same council. Code review gets a different bench than policy work. The chamber lets you assemble the right mix every time — frontier CLI seats sit alongside open-weight API seats without ceremony.

Each seat carries its own role: prosecutor, devil's advocate, junior, principal. The role shapes the prompt, not the model.

  • Frontier CLI: Claude Code, Codex, Gemini
  • API seats: any Anthropic, OpenAI, Google model
  • Open-weight via OpenRouter with tool-loop reading
  • Per-task council, not a fixed lineup

03
The deliberation

Seats reason. Disagree. On the record.

The chamber routes the question through the seats in parallel and then in counter-rounds. Models can read each other's reasoning, push back, change their minds. The transcript captures all of it: timing, tokens, costs, retractions.

This is where the chamber diverges hardest from a chat. There is no smoothing. Dissent is preserved verbatim. A single minority position is a flag, not a bug.

  • Parallel first pass, then counter-rounds
  • Audit-grade transcript with full prompt/response/timing
  • Visual disagreement surfaced, not flattened
  • OpenRouter seats fetch only the files they need

04
The verdict

The Keeper has final say.

The chamber recommends. You ratify, override, or send it back. The AI Conclave never pretends the model vote is the decision — that pretense is what makes most AI tools unsafe for real stakes.

Once you stamp it, the verdict and its full dissent record become a durable artifact. PDF for stakeholders. Markdown for the repo. Plaintext for your records. The decision feeds forward into future sessions.

  • Ratify, override, or remand
  • Verdict carries dissent record forward
  • Exports: PDF · DOCX · Markdown · plaintext
  • Canonical decision records archive automatically

Design principles

Six rules the chamber won't break.

01

Deliberation is not execution.

The chamber reasons. Other tools act. Many agent systems blur this line and regret it. The AI Conclave doesn't.

02

Dissent is a feature.

One model objecting is not an error to suppress. It's a signal the question is harder than a single answer admits.

03

Sandbox is read-only.

Seats can see your project files. They cannot edit them. Mutation is reserved for the Keeper and the tools the Keeper invokes.

04

Provenance is always recorded.

Every session knows where it came from and what it touched. The audit trail isn't a setting. It's the substrate.

05

Local, single-user.

The chamber runs on your machine. No cloud agent platform. No multi-tenant blast radius. Your conversations stay yours.

06

The Keeper holds the gavel.

Models recommend. Humans ratify. Whenever those two roles compete, the human wins by design.
