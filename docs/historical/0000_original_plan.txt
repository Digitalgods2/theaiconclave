AI Switchboard: Full Service Plan
1. Project Summary
AI Switchboard is a background automation service that lets AI coding agents and AI CLI tools consult with one another.
Instead of forcing the user to copy and paste between tools like Codex, Claude Code, Gemini CLI, OpenClaw, Ollama, or local models, Switchboard runs in the background, waits for incoming tasks, routes those tasks through selected AI agents, manages structured review rounds, and returns a controlled final answer.
The system should behave like an automation hub, not a terminal tool.
The user, app, or calling agent submits a task.
One AI acts as the primary agent.
Other AIs act as consultants.
The Switchboard controls the conversation.
The primary agent or calling program gets the final say.
2. Core Concept
User or app sends a task to Switchboard.
Switchboard receives the task, validates it, and decides how to route it.
A primary AI agent analyzes the request first.
One or more consultant agents review the primary agent’s answer.
The consultant agents critique, challenge, or improve the first response.
The primary agent receives the critique and prepares a final answer.
Switchboard packages the result and sends it back to the user or calling program.
The agents do not free-chat with each other. Switchboard controls the flow.
3. Main Product Goals
AI Switchboard should:


Run in the background.


Wait for incoming tasks.


Accept messages from apps, scripts, dashboards, and AI tools.


Route tasks to multiple AI agents.


Support primary and consultant roles.


Support structured debate.


Preserve final authority for the caller or chosen primary agent.


Prevent uncontrolled agent loops.


Protect files and commands with permissions.


Log all task activity.


Expose a dashboard for normal users.


Provide skills so AI apps know how to connect.


4. Background Service Model
AI Switchboard should run as a persistent local service.
Supported environments:
Windows can run it as a background process or Windows service.
macOS can run it as a launchd agent.
Linux can run it as a systemd service.
A server can run it as a FastAPI daemon.
The MVP should use Python, FastAPI, SQLite, subprocess calls, a simple background worker, and a local dashboard.
Docker should not be required.
5. Service Responsibilities
The background service handles task intake, task validation, agent selection, context collection, role assignment, agent execution, consultation rounds, response parsing, disagreement tracking, safety enforcement, final result generation, audit logging, dashboard display, and API responses.
The service should sit in the middle of every exchange. It should not let one agent directly control another agent without the Switchboard seeing and recording what happened.

7. High-Level Architecture
The flow should look like this:
Incoming request enters the FastAPI service.
FastAPI sends it to the task router.
The task router decides task type, primary agent, consultants, and permissions.
The context manager gathers needed project context.
The safety policy checks what the task may access.
The task is saved to SQLite.
A background worker picks up the task.
The orchestrator runs the agent consultation process.
Agent adapters call Codex, Claude Code, Gemini CLI, OpenClaw, Ollama, or other models.
The result builder creates the final answer.
The dashboard and API expose the result.

9. Core Modules
app/main.py
This starts the application.
It should load configuration, initialize the database, register API routes, register dashboard routes, start the background worker, expose a health check, and handle shutdown cleanly.
app/config.py
This loads and validates settings from config.yaml, .env, and environment variables.
Settings should include server host, server port, database path, enabled agents, default primary agent, default consultants, debate round limit, timeout values, safety defaults, dashboard settings, and logging settings.
app/database.py
This manages SQLite setup.
It should connect to the database, create tables, run migrations, provide query helpers, manage task records, manage agent message records, manage result records, and manage approval records.
app/api/tasks.py
This handles task endpoints.
The main routes should include:
POST /api/tasks
GET /api/tasks
GET /api/tasks/{task_id}
POST /api/tasks/{task_id}/cancel
POST /api/tasks/{task_id}/retry
app/api/agents.py
This handles agent discovery and testing.
The main routes should include:
GET /api/agents
GET /api/agents/{agent_name}
POST /api/agents/{agent_name}/test
app/api/approvals.py
This handles user approval for actions.
The main routes should include:
GET /api/approvals
GET /api/approvals/{approval_id}
POST /api/approvals/{approval_id}/approve
POST /api/approvals/{approval_id}/reject
app/services/task_router.py
This decides how a task should run.
It should identify the task mode, select a primary agent, select consultant agents, check required permissions, assign risk level, and decide whether approval will be needed.
Example task modes:
Debug
Code review
Architecture review
Security review
Deployment help
Documentation generation
General consultation
app/services/context_manager.py
This collects context for the agents.
Context may include the user request, selected files, project path, Git diff, logs, error messages, package files, README files, test output, and environment details.
This module should avoid sending entire projects by default. It should send compact, relevant context.
app/services/orchestrator.py
This manages the agent debate.
The default flow should be:
Primary agent receives the task.
Primary agent gives an initial proposal.
Consultant agent receives the task and the primary response.
Consultant critiques the primary response.
Primary agent receives the critique.
Primary agent gives a final answer.
Switchboard saves the final result.
The orchestrator must enforce max rounds. It should never allow endless agent discussion.
app/services/result_builder.py
This creates the final user-facing output.
The final result should include final recommendation, agent agreement level, major disagreements, risks, recommended actions, commands requiring approval, patches requiring approval, and next steps.
app/services/agent_registry.py
This tracks available agents.
It should know which agents are enabled, which commands launch them, which modes they support, whether they are available, and whether they passed a connection test.
app/agents/base.py
This defines the base adapter interface.
Each adapter should support:
name
is_available
run_primary
run_consultant
run_final
test_connection
The rest of the system should call agents through this interface only.
app/agents/codex_adapter.py
This wraps Codex CLI.
It should build prompts, call the Codex command, capture output, parse the result, handle timeouts, handle errors, and return a structured response.
app/agents/claude_adapter.py
This wraps Claude Code.
It should follow the same adapter pattern.
app/agents/gemini_adapter.py
This wraps Gemini CLI.
It should follow the same adapter pattern.
app/agents/openclaw_adapter.py
This wraps OpenClaw.
It should eventually support OpenClaw gateway mode, OpenClaw agent mode, configured model aliases, OAuth-backed providers, and local model routing.
app/agents/ollama_adapter.py
This wraps local Ollama models.
It should support local consultation, cheap review passes, and privacy-sensitive tasks.
app/agents/fake_adapter.py
This is for testing.
It should return predictable fake responses so the Switchboard can be tested without real AI tools.
app/workers/task_worker.py
This runs pending jobs.
For MVP, it can poll SQLite every few seconds.
Basic flow:
Find pending task.
Mark task as running.
Send task to orchestrator.
Save messages as they arrive.
Save final result.
Mark task as completed or failed.
app/safety/policy.py
This defines global safety rules.
Default behavior should deny file writes and command execution unless explicitly allowed.
app/safety/command_guard.py
This blocks dangerous shell commands.
Examples of blocked command patterns:
rm -rf
del /s
format
diskpart
shutdown
reboot
curl pipe bash
sudo
chmod -R 777
app/safety/file_guard.py
This controls file access.
It should limit which folders agents can read, prevent access to secrets by default, block .env unless explicitly approved, block private keys, and log all file access.
app/safety/approval_gate.py
This pauses tasks when an action needs user approval.
Examples:
An agent suggests editing files.
An agent suggests running a command.
An agent suggests installing packages.
An agent suggests deleting or moving files.
An agent suggests changing deployment config.
Switchboard should pause, show the user the proposed action, and wait for approval or rejection.

13. Example Consultation Flow
A dashboard task asks:
Review this Python traceback and suggest the safest fix.
Primary agent:
Codex
Consultant:
Claude Code
Permissions:
Read files only
No writes
No command execution
Switchboard calls Codex.
Codex says the issue is likely a missing dependency.
Switchboard sends Codex’s answer to Claude.
Claude says the missing package is real, but the deeper issue may be the wrong Python environment.
Switchboard sends Claude’s critique back to Codex.
Codex gives a final answer:
Activate the project virtual environment, then install from requirements.txt using python -m pip install -r requirements.txt, then verify pydantic exists in that same interpreter.
Switchboard marks command execution as requiring user approval.
The dashboard shows the final answer and the disagreement.
14. Skills System
Skills are Markdown instruction files that tell AI apps how to connect to Switchboard.
They are not code.
They explain behavior.
A skill tells an AI agent:
How to contact Switchboard.
What endpoint to use.
What JSON format to send.
When to ask for consultation.
How to behave as primary.
How to behave as consultant.
How to return structured results.
What not to do.
15. Skill Types
Generic Switchboard Connector Skill
This tells any AI tool how to contact Switchboard.
It should include the local endpoint, required request format, permission rules, and expected response behavior.
Primary Agent Skill
This tells an AI how to behave when it owns the task.
Primary agent responsibilities:
Make the first proposal.
Read consultant feedback.
Accept or reject critique.
Produce the final answer.
Do not hide disagreement.
Do not claim consensus if there was none.
Consultant Agent Skill
This tells an AI how to behave when it only reviews.
Consultant responsibilities:
Critique the proposal.
Identify bugs.
Identify missed risks.
Suggest better options.
Ask clarifying questions if needed.
Do not take final authority.
Safety Skill
This tells AI agents what they must not do.
Rules:
Do not execute file changes without approval.
Do not request command execution unless allowed.
Do not send secrets.
Do not ask for destructive operations unless absolutely necessary.
Always mark risky actions clearly.
16. Generic Skill File Content
The generic connector skill should say:
Use AI Switchboard when a task would benefit from another AI agent’s review, critique, or alternate reasoning.
Default local endpoint:
http://127.0.0.1:8787/api/tasks
Use Switchboard when:
The user asks to consult another AI.
The problem is complex.
The answer may affect code, files, deployment, security, or data.
A second opinion would reduce risk.
The current agent is uncertain.
Required behavior:
Send a structured task request using the Switchboard Protocol.
Do not send secrets unless the user explicitly allows it.
Do not request file writes unless the user has approved file write permissions.
Do not request command execution unless the user has approved command execution.
If acting as primary:
Make the initial proposal.
Review consultant feedback.
Produce the final answer.
If acting as consultant:
Critique the proposal.
Identify bugs, risks, missing context, and better options.
Do not claim final authority.
Final authority belongs to the calling agent or user-selected primary agent.

21. Safety Model
Safety must be built in from the start.
Default rules:
File reads allowed only inside approved project folders.
File writes disabled by default.
Command execution disabled by default.
Patch application requires approval.
Package installation requires approval.
Network operations require approval.
Dangerous commands blocked.
Secrets should not be sent to agents by default.
All actions should be logged.
Agents should not get unlimited context.
Agents should not get unlimited rounds.
Consultants should not get final authority unless explicitly assigned.

23. Automation Triggers
Automation triggers create Switchboard tasks without manual user entry.
Webhook Trigger
External apps can POST tasks to Switchboard.
Example use:
A build system sends a failed test log to Switchboard.
Folder Watcher
Switchboard watches a folder.
Example use:
When error.log changes, Switchboard analyzes the new error.
Git Watcher
Switchboard watches Git diffs.
Example use:
When files change, Switchboard asks agents to review the diff.
Log Watcher
Switchboard watches logs.
Example use:
When a traceback appears, Switchboard starts a debug task.
Scheduled Task
Switchboard runs a task on a schedule.
Example use:
Every night, review project TODOs and create a report.
For MVP, defer watchers. Start with manual dashboard tasks and HTTP API tasks.
24. Agent Debate Rules
The debate must stay controlled.
Default pattern:
Primary proposal.
Consultant critique.
Primary final.
Optional second round only if needed.
Rules:
Maximum 2 rounds by default.
Consultants critique, they do not decide.
Primary must address consultant risks.
Switchboard records disagreements.
Switchboard does not hide uncertainty.
Switchboard stops if agents loop, repeat, or exceed time limits.
